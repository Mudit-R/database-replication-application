HOW THIS PROJECT WORKS (PLAIN ENGLISH)
=======================================

You don't need to know Java to understand this. Think of this program as a
translator that sits between two databases.

THE BIG PICTURE
----------------
- One database (could be any NoSQL database) sends out a message every time
  a row changes: something was added, changed, or removed.
- Those messages travel through Kafka (think of Kafka as a conveyor belt
  that carries messages from one system to another).
- This program picks each message off the conveyor belt, reads it, and
  figures out the matching SQL command (INSERT / UPDATE / DELETE) needed to
  make the same change happen in a MySQL table.
- It then runs that SQL command against MySQL.

That's it. It's a translator: "NoSQL says this changed" -> "here's the MySQL
command that makes MySQL match."

THE FILES, IN PLAIN TERMS
--------------------------

1. application.yml
   This is the settings file. It's the only file you're likely to ever need
   to change. It says:
     - Where to find Kafka, and which "topic" (channel) to listen to.
     - Where to find the MySQL database.
     - Which table in MySQL to write to.
     - How to find each piece of data inside an incoming message, and which
       MySQL column it belongs in.

2. sample-message.json
   A fake example of a message you might receive. Used only for the demo
   (explained below) so you have something to test with immediately.

3. ReplicationEngineApplication.java
   The "start button" for the real, always-running version of the program.
   You won't need to touch this file.

4. ReplicationProperties.java
   A small helper that reads application.yml into memory when the program
   starts, so the rest of the code can look up your settings.

5. SqlBuilder.java
   The actual "brain" of the translator. Given one message and your
   settings, it works out:
     - which MySQL table to use
     - whether this is an insert/update or a delete
     - which column is the primary key (the special column MySQL uses to
       tell rows apart)
     - the exact SQL command to run
   This file doesn't touch Kafka or MySQL at all - it just does the
   "translate message into SQL" part. That's what makes the demo tool
   possible (see below).

6. ReplicationConsumer.java
   The piece that actually listens to Kafka. For every message it receives,
   it hands the message to SqlBuilder (file #5), gets back a SQL command,
   and runs it against MySQL. This only runs as part of the real,
   always-on application - it needs Kafka and MySQL to actually be there.

7. DemoRunner.java
   A separate, standalone "try it out" tool. It does NOT need Kafka or
   MySQL running. It just reads application.yml and a sample message,
   builds the SQL using the same brain (SqlBuilder), and prints the result
   to your screen so you can see exactly what would happen - safely,
   without touching any real database.

HOW ONE MESSAGE GETS TRANSLATED
---------------------------------
Say this message arrives:

  {
    "header": { "uuid": "INV-8877", "action": "insert" },
    "invoice_amount": 12000.50,
    "vendor": "Google LLC",
    "paid": true,
    "child": { "for_order": "orderid1" },
    "For_order": "parent_orderid1"
  }

The settings file says, in effect: "the value at header.action tells you
the operation, and here's a list of MySQL columns and where their value
lives inside the message." So the program:

  1. Looks at header.action -> sees "insert".
  2. Goes down its list of columns and, for each one, digs into the message
     using the "path" you gave it. A path like "child.for_order" means
     "open the child section, then look for for_order inside it."
  3. Notices that the column called invoice_id has the same path
     (header.uuid) as the special "userkey-path" setting - so invoice_id
     becomes the primary key for this table.
  4. Since the operation is "insert", it builds an "upsert": a command that
     inserts a new row, but if a row with that same primary key already
     exists, it updates it instead. This is the standard, safe way to
     handle both "insert" and "update" messages with one command.
  5. Prints/runs something like:

     INSERT INTO db_invoices
       (invoice_id, invoiceamount, vendor_name, paid_paise, order_for, column1)
     VALUES
       ('INV-8877', 12000.50, 'Google LLC', true, 'parent_orderid1', 'orderid1')
     ON DUPLICATE KEY UPDATE
       invoiceamount = 12000.50,
       vendor_name = 'Google LLC',
       paid_paise = true,
       order_for = 'parent_orderid1',
       column1 = 'orderid1'

If the message's action had been "delete" instead, it would build:

     DELETE FROM db_invoices WHERE invoice_id = 'INV-8877'

WHAT IF A FIELD IS MISSING FROM THE MESSAGE?
----------------------------------------------
No problem - if the program can't find a value at the path you configured,
it just uses NULL for that column instead of failing. The rest of the
message is still processed normally.

TRYING THE DEMO (NO SETUP NEEDED)
------------------------------------
You'll need Java and Maven installed on your computer (these are free,
standard developer tools - if this is your first time, search
"install Java 17" and "install Maven" for your operating system).

Once those are installed, open a terminal inside the project folder and
run:

    mvn compile exec:java

This will:
  - Read application.yml
  - Read the bundled sample-message.json
  - Print out exactly what SQL would be generated

No Kafka, no MySQL, nothing else required - it's completely safe to run.

To try your own message, save it as a .json file somewhere and run:

    mvn compile exec:java -Dexec.args="path/to/your-message.json"

Swap in different values, or a "delete" action, or remove a field, and
re-run it to see how the output changes.

RUNNING THE REAL THING
------------------------
This part does need actual Kafka and MySQL servers running, with the
connection details filled in under application.yml. Once those are ready:

    mvn clean package
    java -jar target/replication-engine-1.0.0.jar

The program will then sit and listen, translating and applying real
messages as they arrive, until you stop it.

A FEW THINGS TO KNOW
-----------------------
- The MySQL table must already exist, with a primary/unique key on the
  column you're using as the primary key (invoice_id in the example).
  This program only writes to existing tables - it doesn't create them.
- If a message fails to translate (e.g. it's missing the action field
  entirely), the real app logs the problem and moves on to the next
  message rather than crashing.
- The "type" hints in application.yml (like VARCHAR(255)) are just notes
  for humans reading the config right now - the program doesn't enforce
  them. MySQL handles converting the incoming values as needed.
